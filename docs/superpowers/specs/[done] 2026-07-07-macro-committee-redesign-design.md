# 委员会宏观分析改版 + 市场广度 + 红涨绿跌 设计文档

- **日期**: 2026-07-07
- **状态**: 待评审
- **范围**: openInvest 委员会宏观分析、宏观快照卡片、市场广度采集、全局宏观判断架构、全局红涨绿跌配色

## 1. 背景与动机

用户希望优化委员会的宏观分析,并增强宏观快照卡片。探索现有实现后,发现需求可归纳为四条主线,且顺带修复两个既有缺陷:

**用户诉求:**
1. 卡片新增市场广度指标:涨-平-跌家数、涨停/跌停家数、赚钱效应(涨幅>3% 个股数量 → AI 判断)、大盘情绪%。
2. 宏观快照从"每个 symbol 内部"抽出,移到持仓摘要下方,**全局只跑一次**;宏观分析师注入宏观数据后,针对各 symbol 行业做轻量敏感度分析。
3. 全局红涨绿跌(A 股惯例)。
4. 完整委员会实况页视觉对齐(已通过 HTML demo 确认)。

**顺带修复的既有缺陷:**
- **B1** 批量运行时,全局宏观判断在每个 symbol 上重复计算 N 次(浪费 + 同批不同 symbol 宏观信号可能自相矛盾)。
- **B2** 市场阶段判定规则依赖 MA20/MA60,但系统从未注入 MA 数据,prompt 用补丁让 LLM"自行推断,不要编造"——规则与数据不自洽。
- **B3** `sh_composite_vol20` 标签写"20日波动率"但值是**年化**波动率(× √252),标签撒谎且孤值无参照,决策价值低。

## 2. 现状(探索结论)

- 宏观分析师是流水线首步(step_index=0),仅跑 R1;数据由 Rust 预取塞进 system prompt,无运行时 tool_call。
- `macro_cache`(SQLite)17 指标,定时任务 `macro_refresh`(交易日 8–22 点每 15 分钟)刷新。
- `MacroSnapshot` 投影 10 指标,挂在 `CommitteeResult.macro_snapshot`,当前渲染在每个 symbol 的 verdict 块底部。
- 前端队列调度器在 store(`_startSymbol`),**每次 invoke 只传一个 symbol**;后端 `run_committee_batch_stream` 的 fan-out 循环实际每次只跑一个 → `portfolio_override` 的 Arc 复用跨 symbol 已失效。
- 已有成熟调度器 `invest/scheduler`:6 段 cron、`requires_trading_day` 自动跳过非交易日、`trigger_cron_job` 手动触发、`try_acquire_job` single-flight 锁。
- 涨停/跌停/上涨/下跌家数已采集;缺平盘家数(数据源已返回未提取)、涨幅>3% 家数(无源)。
- 配色现为"绿涨红跌"(西方惯例),`--color-success`/`--color-error` 同时用于涨跌**和**成功/失败状态。

## 3. 核心架构决策:全局宏观判断对象

**这是本次设计的关键修正**(评审中经反思确定)。全局宏观判断**不应是"委员会运行的副产品"**,而应是与 `macro_cache` 平级、按数据新鲜度刷新的独立全局对象。

### 3.1 问题(旧设计"运行时按 symbol 触发"的缺陷)

宏观快照含两类内容,刷新节奏本就不同:
- **纯市场数据**(上证/北向/涨跌停/涨幅>3%/成交额)—— `macro_cache`,定时任务刷。
- **LLM 判断**(signal/强度/市场阶段/赚钱效应档位)—— 需调用 LLM。

若把 LLM 判断绑在"跑委员会"上,会出现:
- **半残卡片**:没跑过委员会时,判断字段为空,常驻卡片显示一半。
- **时间戳错位**:数据 10:15 刷新、判断还是 10:00 的,同卡数字与结论对不上。
- **赚钱效应分裂**:涨幅>3%家数(随时有)与"火热/冰点"档位(要跑委员会)被割裂。

### 3.2 方案:绑定数据版本的全局判断对象 + 定时任务(交易时段窗口)

- **存储**:新增全局宏观判断对象(persist 到 SQLite),字段含 signal / strength / market_phase / money_effect / money_effect_reason / signal_reason,**外加 `based_on_data_version`(所依据的 macro_cache 数据版本戳)**。
- **新鲜度 = 数据版本一致**:判断有效 ⟺ `based_on_data_version == 当前 macro_cache 最新版本`。底层数据一刷新,判断自动失效。
- **触发(定案:定时任务 + 交易时段窗口 + 手动入口)**:
  - 新增 `macro_verdict` cron job,`requires_trading_day: true`(自动跳过非交易日/时段)。
  - **节奏(用户定案)**:盘前 + 盘中每 30 分钟 + 尾盘,约 10 次/交易日。cron 覆盖 9:15 盘前预热、9:45/10:15…每 30 分钟、14:45 尾盘,排除午休。
  - **手动刷新入口**:复用 `trigger_cron_job('macro_verdict')`,始终可用。
  - **single-flight**:复用调度器现成 `try_acquire_job` 锁,定时与手动天然去重,无需另造锁。
- **委员会运行时**:不再重算全局判断(修复 B1)。各 symbol 只读全局判断做轻量行业敏感度分析。
- **卡片展示**:纯数据随时读 `macro_cache`;判断部分读全局对象,生成中显示占位,不再半残。

### 3.3 收益

卡片不依赖"先跑委员会"、全 symbol 判断天然一致、单 symbol 重跑读缓存不重算、数据与判断永远同版本。代价:一个持久化对象 + 复用现成 cron/锁,无新机制。

## 4. 分层设计

### 4.1 数据层(macro_cache)

- **新增指标 `up_over_3pct_count`**:全 A 涨幅 > 3% 个股数量(赚钱效应基础数据)。数据源见 §7 待办(需新 AkShare provider 或现有活跃度接口扩展)。
- **新增指标 `flat_count`**:平盘家数。数据源已返回,当前未提取,补提取即可。
- **大盘情绪 %**:前端派生 = 上涨 / (上涨 + 下跌) × 100,不新增存储。
- **移除 vol20 展示**(修复 B3):`sh_composite_vol20` 仍可留在 cache 供其他用途,但**不再进宏观快照卡片**;波动率趋势改由 prompt 内部消化(见 4.3)。

### 4.2 全局宏观判断层(新增)

- 新增 `macro_verdict` job + 全局判断对象存储(§3.2)。
- 判断由**独立的全局宏观 prompt**产出(不含任何 symbol 上下文),见 4.3。
- 下游各 symbol 的 gate/流水线统一读全局 `signal`/`strength`(修复 B1)。

### 4.3 LLM Prompt 层

- **拆分为两个 prompt**:
  - `MACRO_GLOBAL_PROMPT`:仅做大盘判断。注入 MA20/MA60 真实数据 + 波动率趋势(修复 B2),用市场广度(涨跌停、涨幅>3%、情绪%)作为**市场阶段 + 赚钱效应的显式判据**。产出 signal/strength/market_phase/money_effect(火热/活跃/平淡/冰点 4 档)/各理由。
  - `SENSITIVITY_PROMPT`:极简。输入 = 全局判断 + 该 symbol 所属行业,输出 sensitivity(positive/negative/neutral) + 一句理由。
- **`emotion_temperature` → `money_effect`**:parser 字段 `emotion_temperature`/… 调整为 `money_effect` + `money_effect_reason`(赚钱效应 4 档),替代原情绪温度语义。

### 4.4 卡片层(前端)

- **宏观快照卡片**:移到持仓摘要下方,**全局展示一次**。分两区:
  - 「宏观指标」:上证 / 北向 / VIX / 金价 / 成交额(5 格,撤 vol20)。
  - 「市场广度」:涨-平-跌(合并格) / 涨停 / 跌停 / 涨幅>3% / 大盘情绪%(5 格)。
  - 底部「赚钱效应」带:档位徽标 + 一句理由。
  - 判断生成中:显示占位/"分析中",纯数据正常显示(§3.2)。
- **per-symbol**:移除整块宏观快照卡,verdict 块内新增**行业敏感度小条**(正向/负向/中性徽标 + 理由)。
- **store**:新增 `globalMacro` 状态(读全局判断 + 手动刷新 action);移除 per-symbol 宏观卡渲染。

### 4.5 配色层(全局红涨绿跌)

**原则:只反转"方向/涨跌/盈亏"语义,保留"状态/动作"语义。**

| 语义类别 | 反转? | 说明 |
|---|---|---|
| 涨/盈利/bullish/risk_on | ✅ → 红 | 新增 `--up` 红 |
| 跌/亏损/bearish/risk_off | ✅ → 绿 | 新增 `--down` 绿 |
| **BUY / SELL 裁决**(用户定案) | ✅ 跟涨跌 | 买入看多=红、卖出看空=绿 |
| Gate 通过/失败 | ❌ 不变 | 状态语义,通过=绿、失败=红 |
| danger/abort 按钮、error 态 | ❌ 不变 | 危险/错误语义 |

- 新增 `--up`/`--down` 语义变量,**不复用** `--color-success`/`--color-error`(后者继续表示成功/失败状态)。
- **审计范围**:`invest-verdict.ts` 硬编码色(BUY/SELL)、`CommitteeLiveTab` 硬编码 hex(总回报/盈亏/资金流向)、`.chip.sig-*` 方向色、portfolio-summary 色 —— 逐一按上表处理。Gate/danger/error 保持不动。

## 5. 数据流(改版后)

```
[定时 macro_refresh 交易日8-22每15min] → macro_cache(纯数据, 带版本戳)
                                              │
[定时 macro_verdict 开盘→收盘每30min] ────────┤ 读最新数据版本
   收盘定版, 有效至次日05:00 ─────────────────┤
   (requires_trading_day, single-flight)      ▼
                                     MACRO_GLOBAL_PROMPT → 全局宏观判断对象
                                     (signal/strength/phase/money_effect + based_on_data_version)
                                              │
                    ┌─────────────────────────┼──────────────────────────┐
                    ▼                          ▼                          ▼
          [宏观快照卡片(全局一次)]     [委员会跑某 symbol]          [手动刷新按钮]
          纯数据读 cache               读全局判断(不重算)           trigger_cron_job
          判断读全局对象               + SENSITIVITY_PROMPT          ('macro_verdict')
                                       per-symbol 行业敏感度
                                              ▼
                                       verdict 块 + 敏感度小条
```

## 7. 技术验证结论(2026-07-07 实测)

### 7.1 miniQMT 作为广度数据源 —— ✅ 验证通过

实测环境:python-runtime 内置 `xtquant` 已安装,QMT 客户端在线。

- `get_stock_list_in_sector("沪深A股")` → 全 A **5204 只**。
- `get_full_tick(全A)` **一次性快照,耗时 0.36s**,0 无效。
- tick 字段含 `lastPrice` + `lastClose` → 可直接算涨跌幅。
- 实测结果:涨幅>3% = 222 只;涨/平/跌 = 639/30/4535。
- **红利**:涨幅>3% + 涨/平/跌/涨停家数**同一次调用全拿到**。→ **修订 §4.1**:`flat_count` 及全部广度指标改为 **miniQMT 同源一次取全**,消除跨源时间戳错位。
- 已有集成:`providers/xtdata.py`(health/kline/realtime_quote)+ `data_source/registry.rs`(Quote 类 miniqmt_on 开关)。新增 provider 方法 `market_breadth()` 走同一 RPC 通道。

### 7.2 macro_cache 版本戳 —— ⚠️ 当前无,需新增

- 表仅逐行 `fetched_at`(每指标独立 upsert),**无全表批次戳**;已有 `is_stale()` 按行龄判断。
- **定案**:`macro_refresh`/广度刷新结束后写入**专门批次标记行**(如 `indicator='_breadth_batch'` 存批次时间戳),全局判断的 `based_on_data_version` 与之比对(见自检 §8-F)。

## 8. 业务/场景逻辑自检(评审第二轮)

将设计代入真实交易日场景走查,发现 5 个需在实现阶段处理的逻辑问题:

- **A. 广度判断的有效期(已定案)**。盘前市场未开,无实时广度;但**收盘广度是当日最终结果,应持续有效到次日 5:00**(接 clearance_convert 5:05 的新交易日边界)。**修正**:① `macro_verdict` **不再盘前跑假广度**,只在开盘→收盘时段跑(9:35 首读→每 30 分钟→14:45 尾盘);② 收盘那次为"今日定版",新鲜度规则泛化现有 `is_stale()` 的周末逻辑——判断有效期延续到次日 05:00 或被新判断替代;③ 5:00→开盘首读之间为"等待开盘"态。
- **B. 赚钱效应分档阈值需校准(已校准,见 §8.1)**。4 档(火热/活跃/平淡/冰点)需阈值。~~原以为早盘家数天然偏少需时段归一化~~——**此假设错误已收回**:A 股集合竞价使高开>3%个股开盘即计入,任意时点家数都是真实快照,无需时段归一化。**定案**:用历史数据校准,以**占比口径**(涨幅>3%家数 / 当日有效家数)定档,消除上市数量逐年增长的漂移。档位**由阈值规则确定性计算**(非 LLM 判定),LLM 仅生成理由句 → 档位一致、可测、省 token。
- **C. miniQMT 可选源降级(中)**。miniQMT 仅在 `miniqmt_on` 时启用;关闭时涨幅>3%无源(tushare 无此指标)。**修正**:降级时赚钱效应显示"数据不足"而非崩;广度指标坚持**同源优先**,不混用 miniQMT+tushare(避免时间戳错位)。
- **D. 敏感度按行业缓存(中,优化)**。敏感度 = f(行业 × 全局 regime),同行业多 symbol 可按 `行业+regime` 缓存,N 次 LLM 降到行业去重次数。
- **F. 版本戳竞态 + 定义(中)**。refresh 与 verdict 若同刻触发,verdict 可能读上一批。**修正**:① cron 错峰(refresh :00/:30,verdict :05/:35);② 版本戳用专门批次标记行(§7.2),绑定广度指标,不猜 `max(fetched_at)`。

**一致性确认(非问题)**:signal(risk_on/off,综合指数/北向/VIX)与 money_effect(纯广度)可合法不一致(如权重股拉指数但广度弱 = risk_on + 平淡),二者并存是有效信息,保留。

### 8.1 赚钱效应分档校准结果(2026-07-07 实测)

脚本:`src-tauri/python-runtime/scripts/calibrate_money_effect.py`(一次性,非运行时依赖)。
样本:miniQMT 全 A 日线,**483 个交易日(2024-07-09 ~ 2026-07-07,覆盖牛熊)**。
口径:`ratio = 涨幅>3%家数 / 当日有效家数`(占比口径,抗上市数量漂移)。

**占比分布**(右偏,均值 11.74% > 中位 8.90%,max 96.46%):P25=5.69% / P50=8.90% / P75=13.98% / P90=20.77% / P95=28.99%。

**分档 vs 次日上证验证**(诚实记录,非单调):

| 档位 | 天数 | 次日上证均涨 | 胜率 |
|---|---|---|---|
| 火热 | 49 | **+0.29%** | **59%** |
| 活跃 | 192 | +0.03% | 55% |
| 平淡 | 119 | +0.09% | 54% |
| 冰点 | 122 | +0.02% | 57% |

**解读(不粉饰)**:
- **火热档是真信号**(+0.29%/59% 明显跑赢),强赚钱效应动量延续成立。
- 中间两档近噪声,冰点胜率 57%(超跌反弹迹象)。→ **赚钱效应本质是"市场状态描述符"**,为委员会提供情绪/广度上下文,**非独立次日择时信号**(符合设计定位)。
- 方法论坦白:验证用"次日上证(权重股加权)",而赚钱效应测广度(中小盘),二者有错配,故此验证偏保守;火热档在错配下仍跑赢反更可信。

**定案分档阈值**(取整贴合分位,规则确定性计算,LLM 仅写理由):

| 档位 | 占比阈值 | 对应分位 | 约合家数@5200 |
|---|---|---|---|
| 冰点 | ≤ 6% | ~P25 | ≤ ~310 |
| 平淡 | 6% – 9% | P25–P50 | ~310–470 |
| 活跃 | 9% – 21% | P50–P90 | ~470–1090 |
| 火热 | > 21% | >P90 | > ~1090 |

阈值为**固定值**(来自 2 年校准,可定期复校),存为常量,非滚动分位(简单可解释)。

## 8.2 定稿前第三轮自检(2026-07-08)

把定稿设计代入实现细节走查,发现 2 个正确性风险(必须实现前解决)+ 3 个定义/澄清项。

- **G. 版本戳覆盖范围不足(正确性风险,已定案)**。§7.2 定的 `based_on_data_version` 只绑广度批次,但 `macro_verdict` 还依赖上证/北向/VIX/MA20/MA60/波动率——这些来自 `macro_refresh`(每 15min),与广度批次(miniQMT,每 30min)**不同刷新源、不同节奏**。若只比对广度戳,会出现"广度新、大盘数据旧"混用,判断依据自相矛盾却显示"新鲜"。**定案**:版本戳绑**统一输入批次戳**——`macro_verdict` 运行时读齐两类数据后写一行 `_verdict_input_batch`(记录本次实际所用广度批次戳 + macro_refresh 批次戳的组合);新鲜度校验对**两个上游戳都比对**,任一上游刷新则判断失效。§7.2 的 `_breadth_batch` 保留作广度子批次戳,`_verdict_input_batch` 是其上层组合戳。
- **H. 手动刷新绕过交易时段门禁(正确性风险,已定案)**。`trigger_cron_job('macro_verdict')` 手动入口**始终可用**(§3.2),但非交易时段 miniQMT `get_full_tick` 返回**上一交易日收盘 tick**(广度=昨收快照),手动一点就产出"假当日判断",且盖掉正确的收盘定版。**定案**:手动刷新**复用与自动一致的时段逻辑**——交易时段内才真跑;非交易时段(盘前/盘后/周末)不跑假广度,直接复用最近收盘定版并提示"非交易时段,显示最近收盘判断"。即手动只提前触发"下一次本该跑的",不制造时段外的假数据。
- **I. 前端需明确四态状态机(完整性)**。§4.4 的"生成中占位"只覆盖一态。实际有四态需区分,避免混淆:① 空态(全新用户无任何判断)→ 引导手动刷新;② 数据不足(miniQMT 关/降级,§8-C)→ 广度区显"数据不足",判断区照常读全局对象;③ 等待开盘(次日 05:00 后至开盘首读之间,§8-A)→ 显"等待开盘",纯数据留最近值;④ 正常。实现时前端按此四态出渲染分支。
- **J. 枚举值需集中定义(待定义)**。`signal`(risk_on/off/neutral)、`strength`(取值域待定,建议 0–100 或 强/中/弱 三档)、`market_phase`、`money_effect`(火热/活跃/平淡/冰点)必须在 parser/prompt 定稿时**列举成闭集**,与前端 chip class(`.sig-*`/`.me-badge.*`,见 demo)**一一对应**,防 LLM 自由发挥导致渲染 fallback。`money_effect` 已闭集(4 档,规则算);其余在实现步骤 4 定稿。
- **K. 敏感度缓存生命周期(澄清)**。§8-D 的"行业+regime 缓存"定为**单次批量运行内的进程内存缓存**(HashMap),regime 变更或换新批次自动失效,**不持久化**——避免跨批次失效竞态,简单且天然随全局判断刷新而重算。

**处置**:G/H 为正确性风险,列入实现步骤 1(H 触及 provider 时段逻辑)与步骤 2/3(G 触及版本戳与 verdict 写入);I 归前端步骤 6;J 归 prompt/parser 步骤 4;K 归编排步骤 5。§9 待办同步补 G/H 为实现前必做项。

## 9. 其他待办(实现阶段解决)

- **[cron 表达式]** `macro_verdict` 精确 cron 需排除午休(11:30–13:00)且相对 refresh 错峰。分段式(如 `0 5,35 9,10,13,14 * * 1-5` + 补边界)实现时定稿并加测试。
- **[i18n]** 新增字段(赚钱效应档位、市场广度标签、敏感度)需补 i18n key(中英)。
- **[Tauri 约束]** 定时任务仅在 app 运行时生效;app 关闭期间不跑,重启后按 cron 下一 tick 恢复。首次/失效靠手动刷新兜底。
- **[空态]** 全新用户无任何数据时,宏观卡显示空态 + 引导手动刷新,不崩(并入 §8.2-I 四态)。
- **[实现前必做]** G 统一输入批次戳 + H 手动刷新时段门禁(§8.2),两者为正确性风险,不是可选优化。

## 10. 实现工作分解(概览,非承诺顺序)

1. 数据层:miniQMT `market_breadth()` provider 方法(涨幅>3% + 涨/平/跌 同源)+ 降级策略 + **手动/自动统一时段门禁(§8.2-H)** + 批次版本戳。
2. 存储层:全局宏观判断对象表 + 读写 + **`_verdict_input_batch` 统一戳(§8.2-G)**。
3. 调度层:`macro_verdict` job 定义 + dispatch 分支 + cron(错峰 + 仅开盘→收盘,不盘前跑假广度)+ **verdict 写入时双上游戳比对(§8.2-G)**。
4. Prompt/parser:拆分 global/sensitivity prompt、`money_effect` 字段改造(含分档阈值)、注入 MA/波动率、**枚举闭集与前端 chip class 对齐(§8.2-J)**。
5. 委员会编排:去除 per-symbol 宏观重算,接入全局判断 + 敏感度(**单次运行内存缓存,§8.2-K**)。
6. 前端:全局宏观快照卡片(两区+赚钱效应带)、敏感度小条、store `globalMacro`、手动刷新、**四态状态机(§8.2-I)**。
7. 配色:`--up`/`--down` 变量 + 全局红涨绿跌审计替换。
8. 测试:cron 表达式、**双上游新鲜度失效(§8.2-G)**、**非交易时段手动刷新不产假判断(§8.2-H)**、parser 新字段、配色回归、降级路径。
