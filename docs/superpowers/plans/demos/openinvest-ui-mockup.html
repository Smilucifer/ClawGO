改动 2b (v2 新增)：买入 / 观望 / 调整持仓 对话框

v2 关键架构补充（Q4 决议）— 资产分两类：

类型含义占用现金?影响 PnL?进入命中率?
HOLD 真实持仓（已买入） ✅ 扣减 cash ✅ 真实 PnL ✅ 计入
WATCH AI 推荐后观望（未真实买入） ❌ 不动 📊 虚拟 PnL（按名义金额） ✅ 计入（避免幸存者偏差）

为什么 WATCH 也要进命中率： AI 推荐 100 只股票,你只买了 10 只。如果只统计 HOLD 的命中率,就只验证了"你愿意买的子集", 另外 90 只 WATCH 的命中率统计才是验证 AI 推荐质量本身的关键 — 否则会出现严重的幸存者偏差,看起来 AI 很准但其实是你筛选过的。

v2 新增 — 对话框设计要点：

1. 操作分离：买入(HOLD) / 加入观望(WATCH) / 调整持仓 是三个独立入口,不再混在「调整持仓」一个按钮里。
2. WATCH 用名义金额：默认按金额而非股数,避免不同 symbol 因股价差异(贵州茅台 ¥1700+ vs 中国平安 ¥45)导致命中率 weighting 失真。
3. WATCH 关联裁决：必填一个委员会裁决（或显式标记不关联）— 这是后续 verdict_review 命中率回查的 join key。
4. 调整持仓：4 选 1 操作（加仓/减仓/修改成本/转 WATCH），UI 实时显示加权平均成本计算公式。
5. WATCH→HOLD 关键选择：保留 WATCH 命中率（推荐）vs 归并到 HOLD —前者更诚实,后者会虚高 PnL,默认推荐保留。

Schema 影响：
CREATE TABLE holdings (
  symbol TEXT, qty REAL, avg_cost REAL, currency TEXT,
  kind TEXT NOT NULL DEFAULT 'hold', -- 'hold' | 'watch'
  entry_date TEXT NOT NULL, -- 入场/观望日
  notional REAL, -- WATCH 名义金额
  linked_verdict_id TEXT, -- 关联裁决(WATCH 必填)
  status TEXT DEFAULT 'active', updated_at TEXT,
  PRIMARY KEY (symbol, currency, kind)
);

v2 修订（Q4 决议） — 多资产并发 + 资产内串行：

并发模型：
• 5 个 symbol 完全并发独立运行（每个 symbol 一个 tokio task,有独立 cancellation token）
• 每个 symbol 内部 LLM 调用串行（Macro → Q1 → R1 → W1 → Q2 → R2 → CIO）— 一次只有 1 个 LLM 在该 symbol 内对话
• 全局 LLM 并发上限 8（D8）保护 Provider API,5 symbol × 1 并发 = 5,未触上限

为什么这样设计：
• 资产内串行 → token 流可读性高,调试时能清楚看到当前哪个角色在说话
• 资产间并发 → 总耗时 ≈ 最慢 symbol（约 30-60s）,而非 5× 叠加
• 失败隔离 → 一个 symbol 失败不影响其他 symbol（独立 task + cancellation token）

UI 设计：
• 顶部总览表格：每行一个 symbol,展示当前阶段 + 7 个微型角色徽章 + tokens + verdict
• 中部 Tab 切换器：点击切换到单资产详情
• 底部详情卡片：选中资产的内部串行执行历史（已完成步骤折叠为一行,当前角色展开 streaming token 流）
• 「全部取消」按钮：cancel 所有 symbol 的 cancellation token
• 单资产「⏸」：仅取消该 symbol 的 task,其他继续

Rust 实现要点：
• tokio::spawn 每个 symbol 一个 task + CancellationToken::child_token()
• tokio::join! 等待所有 symbol 完成
• 资产内串行：直接 await 链,不用 join_all
• LlmGovernor Semaphore(8) 全局共享,在每次 HTTP 调用前 acquire
